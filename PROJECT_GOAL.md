# Project Goal

## 과제명

Kubebuilder 기반 Operator 개발 자동화를 위한 AI 기반 생성·검증 시스템 구축

## 과제 유형

AI 시스템/솔루션 개발

## 대상

- Kubebuilder 기반 Kubernetes Operator를 설계·개발하는 내부 개발자 및 플랫폼 엔지니어
- CRD 정의, Controller 구현, RBAC/Manifest 작성, 테스트 및 배포 검증까지 수행하는 개발 업무
- GitHub, Jenkins, Harbor, Argo CD 기반 개발·검증·배포 체계를 사용하는 환경

## 문제 정의

Kubebuilder 기반 Kubernetes Operator 개발은 CRD 정의, Controller 구현, RBAC/Manifest 작성, 테스트, 배포 검증까지 여러 단계를 순차적으로 수행해야 하는 개발 업무입니다.

실제 개발 과정에서는 어떤 산출물을 어떤 순서로 작성해야 하는지, 각 단계에서 무엇을 검증해야 하는지 한 번에 파악하기 어려운 경우가 많습니다. 특히 초기 개발자나 Kubebuilder 경험이 적은 개발자는 개발 절차 전반을 스스로 설계하고 진행하기 어렵고, 중간 단계에서 시행착오와 반복 수정이 발생하기 쉽습니다.

또한 `generate`, `manifests`, 빌드, 테스트, 이미지 생성, 배포 검증 단계에서 오류가 발생하면 원인 분석과 다음 조치 방향을 파악하는 데 추가 시간이 소요됩니다. 이로 인해 개발 과정이 개인 경험과 숙련도에 의존하게 되고, 산출물 품질, 검증 통과율, 개발 속도에 편차가 발생합니다.

## Pain Point

- 어떤 산출물을 어떤 순서로 작성해야 하는지 명확하지 않아 초기 개발 단계에서 자주 막힙니다.
- 단계별로 무엇을 검증해야 하는지 판단하기 어려워 시행착오와 반복 수정이 발생합니다.
- `make generate`, `make manifests`, 빌드, 테스트, 배포 검증 과정에서 오류가 발생하면 원인과 다음 조치 방향을 파악하는 데 시간이 많이 소요됩니다.
- 개발 과정이 개인 경험과 숙련도에 의존하여 생산성, 산출물 품질, 검증 통과율에 편차가 발생합니다.

## 개선 목표

- Kubebuilder 개발 절차를 단계적으로 안내하여 수행 순서를 쉽게 이해할 수 있도록 합니다.
- 필수 산출물 생성을 지원하여 초기 개발 부담을 줄입니다.
- 테스트 및 배포 검증까지 연계하여 실제 활용 가능한 결과물 확보를 지원합니다.
- 오류 발생 시 원인과 해결 방향을 제시하여 시행착오와 수정 공수를 줄입니다.
- 개발자 숙련도에 따른 절차 이해도와 산출물 품질 편차를 완화합니다.

## 평가 지표

| 지표 | 목표 |
| --- | --- |
| 개발 단계별 작업 시간 단축 | 요구사항 입력부터 주요 산출물 생성까지 기존 수작업 대비 50% 이상 단축 |
| 필수 산출물 생성 완성도 | CRD, Controller, RBAC/Manifest, 테스트 초안 등 필수 산출물 생성 충족률 90% 이상 |
| 검증 단계 통과율 | `make generate`, `make manifests`, 빌드, 테스트 등 검증 단계의 1차 통과율 80% 이상 |
| 오류 대응 지원 속도 | 오류 발생 후 원인 및 해결 방향 제시까지 평균 10분 이내 |

## 개선 방법

- 자연어 기반 개발 요구사항을 입력받아, AI Agent가 Kubebuilder 기반 Operator 개발 절차를 단계별로 해석하고 수행 순서를 안내합니다.
- RAG를 활용하여 Kubebuilder 가이드, 사내 예제 코드, YAML 템플릿 등 관련 지식을 검색하고, 이를 바탕으로 내부 스펙을 구성합니다.
- 내부 스펙을 기준으로 CRD, Controller, RBAC, Manifest, 테스트 초안 등 단계별 산출물을 순차적으로 생성합니다.
- MCP 기반 GitHub 연계를 통해 기존 유사 구현과 프로젝트 구조를 참조하고, 생성 결과를 실제 개발 흐름에 반영할 수 있도록 합니다.
- Jenkins와 연계하여 `make generate`, `make manifests`, 빌드, 테스트 결과를 수집하고, 오류 발생 시 AI Agent가 원인과 해결 방향을 제시합니다.
- Harbor 및 Argo CD와 연계하여 이미지 저장, 배포 반영, 배포 결과 확인까지 이어지는 흐름을 구성합니다.
- 이를 통해 개발자가 무엇을 어떤 순서로 수행해야 하는지 단계적으로 안내받고, 중간에 막히는 경우에도 해결 방향까지 지원받을 수 있도록 합니다.

## 활용 계획

| 항목 | 내용 |
| --- | --- |
| 로컬 LLM 기반 요구사항 해석 | 자연어 요구사항을 Semantic Parsing하여 리소스 구조, Spec/Status 필드, Reconcile 책임, 권한 범위, 배포 단위를 포함하는 구조화 스펙으로 변환 |
| RAG 기반 유사 사례 검색 | Kubebuilder 가이드, 사내 예제 코드, YAML 템플릿, 테스트 샘플을 내부 지식베이스와 Vector DB로 구성하고 요구사항 기반 검색 수행 |
| Reranker 기반 컨텍스트 최적화 | 검색 결과를 재정렬하여 도메인 적합도가 높은 참조 문서만 생성 단계의 참고 정보로 활용 |
| Few-shot 기반 생성 유도 | 유사 요구사항과 산출물 예시를 Few-shot 형태로 반영하여 생성 형식의 일관성과 정확도 향상 |
| LangChain 기반 Agent Workflow | 요구사항 해석, 검색, 산출물 생성, 검증, 오류 분석 단계를 연결하여 순차적인 개발 흐름 자동화 |
| LLM 기반 산출물 생성 | 구조화 스펙과 참조 문서를 바탕으로 CRD, Controller, RBAC, Manifest, 테스트 초안을 단계적으로 생성 |
| MCP 기반 GitHub/Jenkins 연계 | 유사 구현 조회, 브랜치/commit/push/PR 초안 생성, Jenkins 검증 결과 수집과 후속 조치 연결 |
| LLM 기반 오류 분석 및 보정 | 검증 로그를 해석하여 오류 원인을 분류하고 수정 포인트와 해결 방향을 제시하며 필요 시 해당 산출물만 부분 재생성 |

## 최적화 계획

- 요구사항 해석, 산출물 생성, 오류 분석 목적에 적합한 로컬 LLM 모델을 선정하고 단계별 입력 범위를 제한하여 추론 성능과 응답 품질을 최적화합니다.
- Kubebuilder 가이드, 사내 예제 코드, YAML 템플릿, 테스트 샘플을 지속적으로 확장하고 검색 인덱스와 질의 구성을 개선합니다.
- Reranker로 도메인 적합도가 높은 문서만 선별하여 불필요한 컨텍스트를 줄이고 생성 품질과 응답 효율을 높입니다.
- LangChain Workflow를 요구사항 해석, 검색, 생성, 검증, 오류 분석 단계로 분리하여 실패 지점을 명확히 하고 필요한 단계만 재수행할 수 있도록 구성합니다.
- GitHub 반영, Jenkins 검증, 로그 수집을 자동화하여 생성 결과가 실제 개발 흐름과 자연스럽게 연결되도록 합니다.
- 검증 과정에서 발생한 오류를 단계별·유형별로 분류·축적하고, 해당 산출물만 부분 보정 또는 재생성하여 수정 공수를 줄입니다.
- 축적된 오류 유형은 프롬프트, 검색, 생성 규칙 개선에 반영하여 생성 품질을 지속적으로 높입니다.

## 핵심 차별점

본 프로젝트는 단순 코드 생성기가 아닙니다.

핵심 차별점은 생성, 검증, 오류 분석, 수정 방향 제시가 이어지는 실행형 개발 Agent라는 점입니다. 따라서 모든 산출물은 실제 로컬 개발환경에서 실행·검증 가능한 흐름을 기준으로 작성되어야 합니다.

## 1차 목표

1차 MVP에서는 로컬 실행형 Agent 흐름의 가능성을 검증합니다.

- 로컬 개발환경 점검 절차 수립
- Kubebuilder scaffold 생성 절차 정리
- 예시 API 생성 절차 정리
- 필수 검증 명령 실행 흐름 정리
- 실패 로그 분석 기준 수립
- 향후 RAG, MCP, CI/CD 연계가 가능한 구조 설계
